  瓶颈分析:
  lsr z_src_hi.b, z_src.b, #4（第106行）占 62.06%
  采样，这不是因为移位操作本身慢，而是因为它是 ld1b
  加载结果的第一个消费者（按字节视图 .b），ARM SVE 的 ld1b 到 lsr .b 存在
  load-use 延迟（通常 4-7 个周期），CPU 在此指令处流水线停顿等待数据就绪。and
  指令虽然也依赖 ld1b，但它使用
  .d（双字）视图，在部分微架构上可能通过不同执行端口与 lsr
  并行发射，采样因此归到了 lsr 上。

  核心问题：循环体过短，每次迭代只有一条加载 +
  一组计算，无法有效隐藏加载延迟。load-use 延迟的空转周期占比过大。

  优化建议:

  - 建议一：循环展开 + 软件流水线（预期提升 30-50%）
  展开 2-4 次迭代，在当前迭代的 ld1b 和 lsr 之间插入下一次迭代的
  ld1b（预加载），使加载延迟与当前迭代的计算（tbl/eor/st1b）重叠。关键改动：
  // 第1次迭代的加载提前到循环开头
  ld1b    z_src0.b, p0/z, [x_src, x_pos]
  .Lloop_unrolled:
      // 处理第1次迭代的同时，加载第2次迭代的数据
      lsr     z_src0_hi.b, z_src0.b, #4
      and     z_src0_lo.d, z_src0.d, z_mask0f.d
      ld1b    z_src1.b, p0/z, [x_src, x_pos_next]  // 预加载！
      tbl     ...
      eor     ...
      st1b    ...
      // 第2次迭代：lsr 此时数据已就绪
      lsr     z_src1_hi.b, z_src1.b, #4
      ...
  - 为什么能提升：将 load-use 延迟隐藏在上一迭代计算指令的执行时间内。
  - 建议二：使用 SVE 预取指令 prfm / prfwd（预期提升 5-15%）
  在循环开头插入 SVE 预取指令，提前将下一轮迭代的数据拉入 L1 缓存：
  prfwd   p0, [x_src, x_pos, mul vl]  // 预取下一个向量长度的数据
  - 建议三：考虑 ld1b 加载后先做 and 再做 lsr 的微调（预期提升不确定，需实测）
  虽然两者都依赖 ld1b，但调整指令顺序可能影响微架构调度。在部分 ARM 核心上，.d
  视图的 and 可能在 lsr .b 之前获得操作数。

  架构级建议

  1. 循环展开 + 指令重排（ARM SVE
  优化核心方向）：当前循环体过于紧凑，单次迭代的加载延迟无法被计算指令覆盖。通过
