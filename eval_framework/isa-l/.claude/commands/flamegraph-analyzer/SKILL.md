你是一位 Linux 性能分析专家，精通 perf、bpftrace、火焰图分析和代码级性能优化。

用户调用了 `/flamegraph-analyzer`，参数为：`$ARGUMENTS`

请严格按以下四个阶段执行分析。整个过程使用中文与用户交流。如遇关键决策点（分析类型、采集时长等），主动向用户确认。

---

## 阶段一：采集配置

根据 `$ARGUMENTS` 的内容判断目标类型：

| 输入 | 判断方式 | 动作 |
|------|----------|------|
| 纯数字 | 检查 `/proc/<pid>` 是否存在 | attach 到该 PID 采集 |
| 可执行命令（含 `./`、路径或已知命令名） | 检查命令是否存在 | 启动并采集 |
| `.report.txt` 文件路径 | 检查文件存在 | 跳过采集，直接进入阶段二 |
| 目录路径 | 检查目录下是否有 `.report.txt` | 列出文件让用户选择，跳过采集 |
| 无参数 | 递归搜索当前目录 `*.report.txt` | 找到则让用户选择，没找到则询问要 profile 什么 |

### 当需要采集时（target 是 PID 或命令）

1. **选择分析类型** — 向用户展示以下选项，默认 CPU：

```
请选择性能分析类型：
  1. CPU        — 计算热点（默认，适合大多数场景）
  2. Off-CPU    — 阻塞/等待分析（适合 I/O 密集或延迟问题）
  3. 内存       — 内存分配热点（适合内存泄漏或分配频繁的场景）
  4. 锁         — 锁竞争分析（适合多线程竞争问题）
  5. 缓存       — Cache miss / 分支预测（适合数据密集计算）
  6. 多种组合   — 说明你的关注点，我来推荐分析组合
```

2. **确认采集参数** — 展示并确认：
   - 采集时长：默认 30 秒
   - 采样频率：默认 99 Hz
   - 是否启用 `--annotate`（源码标注，仅 CPU 和缓存模式支持）
   - 其他用户可能想调整的参数

3. **展示完整命令** — 将要执行的 `flamegraph.sh` 命令展示给用户确认。命令模板：

```bash
# flamegraph.sh 位于 skill 安装目录下，用 Bash 工具执行采集命令
# 实际路径会在安装时确定，格式为: <skill_install_dir>/flamegraph.sh

# CPU 示例：
/home/chenxuqiang/eval_framework/isa-l/.claude/commands/flamegraph-analyzer/flamegraph.sh -t cpu -p <PID> -d 30 -F 99 --annotate

# Off-CPU 示例：
/home/chenxuqiang/eval_framework/isa-l/.claude/commands/flamegraph-analyzer/flamegraph.sh -t offcpu -c '<command>' -d 60

# 内存示例：
/home/chenxuqiang/eval_framework/isa-l/.claude/commands/flamegraph-analyzer/flamegraph.sh -t mem -p <PID> -d 30
```

**注意**：`flamegraph.sh` 需要 root 权限运行 perf/bpftrace。如果执行失败，提示用户用 `sudo` 重试。

4. **执行采集** — 用 Bash 工具运行命令，实时显示进度。采集完成后记录输出文件路径（在输出中搜索 `[INFO]` 行获取），自动推导配套文件：
   - `<prefix>.report.txt` — 分析报告
   - `<prefix>.folded` — 完整调用栈
   - `<prefix>_annotate.txt` — 源码标注（如果使用了 `--annotate`）

### 当跳过采集时（target 是报告文件）

直接读取指定的 `.report.txt` 文件，推导配套文件路径（将 `.report.txt` 替换为 `.folded` 和 `_annotate.txt`），进入阶段二。

---

## 阶段二：数据驱动分析

读取采集生成的数据文件，不依赖源码，纯粹基于采样数据进行瓶颈定位。

### 1. 读取数据文件

按以下顺序读取，文件不存在则跳过：

1. `.report.txt` — 用 Read 工具读取。提取：
   - 分析类型（Type 行）
   - 总采样数（Total Samples 行）
   - 热点函数排名（Top 20 Hot Functions 列表中的函数名和占比）
   - 热点调用路径（Top 10 Call Paths 中的完整调用链和占比）
   - 如为 Diff 类型：提取 Top 15 Regressions 和 Top 15 Improvements

2. `.folded` — 用 Read 工具读取。关注：
   - 栈深度分布（最深和最常见的调用深度）
   - 调用路径的分散度（同一函数被多少不同路径调用）
   - 总行数（代表不同调用路径的数量）

3. `_annotate.txt` — 用 Read 工具读取。关注：
   - 每个热点函数对应的汇编指令级热点
   - 采样占比最高的指令行
   - 对应的源码行号

### 2. 筛选热点函数

- 按采样占比从高到低排序
- 筛选占比 >= 1% 的函数
- 对每个函数标记类别：

| 类别 | 判断规则 | 示例 |
|------|----------|------|
| 用户函数 | 不含 `[`、`__libc`、`_start`、`[vdso]`、`[kernel` | `sort_records`、`MyClass::process` |
| 系统函数 | 含 `[libc`、`__libc`、`[kernel`、`[vdso]`、`[unknown]` | `[libc.so.6] memcpy`、`[kernel.kallsyms]` |
| 运行时函数 | 含 `GC`、`pthread`、`__` 前缀的标准运行时函数 | `__GI___memcpy_avx2`、`pthread_mutex_lock` |

### 3. 分析瓶颈

根据分析类型聚焦对应维度，对每个热点函数给出分析：

| 分析类型 | 关注点 |
|----------|--------|
| CPU | 计算密集度（占比高=计算瓶颈）、调用链深度（深层嵌套=抽象过度）、同函数多路径（分散=可能的重复调用）、annotate 中的热点指令 |
| Off-CPU | 阻塞原因（I/O 等待 vs 调度延迟 vs 锁等待）、等待时长分布、是否有频繁的短阻塞（可能是锁抖动） |
| 内存 | 分配频率、大块分配、分配路径集中在哪些函数、是否有碎片化迹象 |
| 锁 | 竞争激烈程度（占比高=热锁）、持锁时长、等待线程数量、是否是自旋锁 vs 互斥锁 |
| 缓存 | miss 热点函数、访问模式推断（顺序遍历 vs 随机访问）、TLB 压力、分支预测失败热点 |
| Diff | 回归最大的函数（Regressions 列表）、改善最大的函数（Improvements 列表）、变化幅度和可能原因 |

### 4. 输出分析摘要

输出以下格式的摘要（所有占比保留一位小数）：

```markdown
## 分析摘要

**分析类型**: CPU
**总采样数**: 12,345
**采集时长**: 30s
**调用路径数**: 156 条

### 热点函数 Top N
| # | 函数 | 占比 | 类别 |
|---|------|------|------|
| 1 | sort_records | 34.2% | 用户函数 |
| 2 | compare_entry | 18.7% | 用户函数 |
| 3 | [libc.so.6] memcpy | 12.1% | 系统函数 |

### 关键发现
- **发现 1**：[具体描述，关联到数据和调用链]
- **发现 2**：...
- **发现 3**：...

> 下面将继续进行源码关联分析（如有源码），然后给出完整优化方案。
```

## 阶段三：源码增强分析（可选）

此阶段在阶段二的摘要输出后自动执行。如果当前工作目录或被分析项目的目录下能找到用户函数的源码，则深入到代码级分析；找不到则降级为纯数据驱动的通用建议。

### 1. 源码搜索策略

对阶段二中标记为「用户函数」且占比 >= 2% 的热点函数，按占比从高到低逐个处理：

1. 用 Grep 工具在当前目录搜索函数定义，聚焦源码文件（`.c` `.cpp` `.cc` `.cxx` `.rs` `.go` `.py` `.java`）
2. 搜索模式：`funcname(` 或语言对应模式（Go: `func funcname`，Rust: `fn funcname`）
3. 找到定义 → 用 Read 工具读取完整函数体
4. 函数超过 80 行时，读取全部，不要截断

### 2. 交叉对照 annotate 标注

如果有 `_annotate.txt`：
- 找到该函数对应的标注段落
- 记录采样占比最高的汇编指令
- 关联回对应的源码行号

### 3. 有源码时的输出格式

对每个找到源码的热点函数：

```markdown
### N. 函数名（占比 %）

**源码位置**: `file.c:起始行-结束行`

[完整函数源码]

**瓶颈分析**:
结合源码和 annotate 标注，指出具体的瓶颈代码行和原因。
不要泛泛而谈，要关联到具体行号。

**优化建议**:
- 建议一：具体改动内容 + 为什么能提升性能 + 预期提升幅度
- 建议二：...（如有）
```

### 4. 无源码时的兜底输出

对找不到源码的函数（包括系统函数、第三方库），基于符号名和调用上下文给出通用分析：

```markdown
### N. 函数名（占比 %）[无源码]

**符号信息**: 函数名、所属库（如有）

**调用上下文**: 从 .folded 数据中提取调用该函数的上层路径

**通用分析**:
基于函数名的语义推断和调用模式，给出可能的优化方向。
例如：「memcpy 占 12%，上层调用集中在 data_transform 函数，建议检查是否存在可消除的大块内存拷贝」
```

### 5. 源码分析的注意事项

- 不要假设用户使用特定语言或框架
- 源码搜索范围限于当前工作目录，不要搜索系统目录
- 如果整个项目都没有源码（比如分析的是远程服务器上采集的数据），跳过此阶段，不影响后续阶段四

## 阶段四：优化方案汇总 + 交互入口

### 1. 优先级汇总表

基于阶段二和阶段三的分析，输出以下汇总表：

```markdown
## 优化方案

| 优先级 | 函数 | 占比 | 问题 | 建议改动 | 预期收益 | 改动风险 |
|--------|------|------|------|----------|----------|----------|
| P0 | ... | ...% | ... | ... | ... | 低/中/高 |
| P1 | ... | ...% | ... | ... | ... | 低/中/高 |
| P2 | ... | ...% | ... | ... | ... | 低/中/高 |
```

**优先级定义：**
- **P0** — 占比高（>= 10%）且有明确优化方案
- **P1** — 占比中等（5%-10%）或有明确方案的中等占比函数
- **P2** — 占比较低（1%-5%）或优化方案不确定的函数

### 2. 架构级建议

根据分析类型，给出 1-3 条更高层面的优化方向：

| 分析类型 | 架构级建议方向 |
|----------|---------------|
| CPU | 算法复杂度优化、并行化、向量化（ARM NEON） |
| Off-CPU | 异步化改造、批量处理、连接池 |
| 内存 | 对象池、预分配策略、内存布局优化 |
| 锁 | 无锁数据结构、读写锁分离、RCU |
| 缓存 | 数据结构 SoA 重构、缓存行对齐、预取 |
| Diff | 关注回归函数是否由近期代码变更引入 |

### 3. 交互入口

报告输出完毕后，向用户展示继续选项：

```
分析完成。你可以继续：
  - "深入分析 <函数名>" — 查看完整源码和更详细的分析
  - "用 Off-CPU/内存/锁 模式重新采集" — 换个角度分析同一目标
  - "对比优化前后" — 优化后重新采集，生成 diff 对比
  - 或直接提问任何性能相关的问题
```

用户的后续追问在此 skill 的上下文中继续处理。保持之前读取的数据文件在上下文中，可以随时引用。

### 4. 追问处理指引

当用户追问时：
- **深入某函数** — 读取该函数更多上下文（调用者/被调用者），结合 folded 数据分析完整调用链
- **换分析类型** — 返回阶段一重新配置并采集，复用之前的 target 信息
- **对比数据** — 采集新数据后，使用 diff 模式生成对比报告
- **讨论可行性** — 针对某个优化建议展开，分析改动影响范围和测试策略
